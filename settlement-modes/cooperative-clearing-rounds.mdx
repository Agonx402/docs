---
title: "Cooperative Clearing Rounds"
description: "How one signed roster can advance many lanes and apply only residual balance changes."
---

Cooperative clearing rounds are the densest settlement path in Agon V4.

Instead of each lane settling independently, a group of participants signs one shared `agon-round-v4` message. That round advances many lane targets at once and updates only the participant balances that still need to move after the included payments cancel each other out.

## Overview

Suppose Alice owes Bob, Bob owes Carol, and Carol owes Alice.

If every lane settles independently, the protocol still touches each participant balance for each edge.

If all three sign one clearing round, the protocol can:

1. move each lane's cumulative amount forward
2. compute the net effect on each participant
3. apply only the residual balance changes that remain

That is what multilateral clearing means in Agon.

## What Is Signed

`agon-round-v4` signs:

1. the deployment message domain
2. one shared token id
3. the participant roster
4. each participant block's outgoing lane targets

Each block names the payer by `participant_id`. Each entry inside the block points to the payee by `payee_ref`, which is an index into the signed roster.

## What The Program Verifies

When `settle_clearing_round` runs, the program checks:

1. the Ed25519 instruction is present and self-contained
2. every signer signed the exact same round message
3. the number of signers matches the participant roster length
4. each participant account is the canonical PDA for the participant block it represents
5. each channel account is the canonical `channel-v2` PDA for the payer, payee, and token it represents
6. every target cumulative amount moves forward
7. every participant with a negative final position has enough balance to cover it

Only after those checks pass does the program write the new lane states and apply the net participant balance changes.

## What Changes Onchain

A clearing round updates two kinds of state:

### Channel state

Every included lane gets a new `settled_cumulative`. If the lane had locked balance, the round consumes that first.

### Participant balances

After all lane deltas are known, the program computes the net change per participant and applies only those residual changes.

## Example

```ts
const message = createClearingRoundMessage({
  tokenId: 1,
  blocks: [
    {
      participantId: aToB.channel.payerId,
      entries: [{ payeeRef: 1, targetCumulative: nextA }],
    },
    {
      participantId: bToC.channel.payerId,
      entries: [{ payeeRef: 2, targetCumulative: nextB }],
    },
    {
      participantId: cToA.channel.payerId,
      entries: [{ payeeRef: 0, targetCumulative: nextC }],
    },
  ],
});
```

## When To Use This Path

Use cooperative clearing rounds when:

- several participants are already coordinating
- they are willing to co-sign one shared round
- there is value in letting payments inside the group cancel each other out first

Today this path is already useful under Ed25519. Aggregate signatures can make it denser later, but V4 does not depend on that future work to be meaningful.
