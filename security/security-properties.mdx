---
title: "Security Properties"
description: "The protocol checks in V4 that matter most to builders."
---

Agon V4 tightened the parts of the protocol that matter most for permanent identities, permanent channels, and offchain signed settlement.

This page covers the checks builders usually care about most.

## Signed Domain Separation

Every signed message includes `message_domain`.

In V4, that value is derived inside the program from:

1. a fixed Agon domain tag
2. the program id
3. the chain id

That prevents deploy-time tooling mistakes from silently choosing a bad signing domain.

## Distinct Chain IDs Per Environment

Mainnet, devnet, testnet, and localnet all use different chain ids in V4.

That means the derived `message_domain` changes across environments. A message signed for devnet should not verify on mainnet, and vice versa.

## Permanent Participants And Permanent Channels

V4 removed participant recycling and channel close-and-reopen flows.

That removes a whole class of replay complexity:

- participant ids do not get reused
- there is no reopened payment channel for an old signed message to target

## Canonical PDA Checks

Settlement paths do not trust arbitrary program-owned accounts just because they deserialize correctly.

V4 verifies canonical participant PDAs and canonical `channel-v2` PDAs. This matters most in bundle settlement and cooperative clearing rounds, where several accounts arrive through `remaining_accounts`.

## Monotonic Cumulative Settlement

Both unilateral commitments and cooperative rounds must move lane state forward.

If a signed message tries to settle to a cumulative amount that is less than or equal to the current settled amount, the protocol rejects it.

## Timelocked Channel Maintenance

V4 puts a timelock in front of actions that would otherwise surprise the other side of the relationship:

1. unlocking lane-specific locked funds
2. rotating the channel signing key

These actions require a request step and an execution step separated by the configured timelock.

## Self-Channel Rejection

V4 explicitly rejects self-channels.

That prevents a payer from creating a channel to itself and using settlement to bypass the unlock timing model.

## Summary

Taken together, these checks mean:

1. signed messages are bound to one deployment
2. account lookups are bound to canonical state
3. settlement only moves forward
4. channel maintenance cannot surprise the other side instantly
