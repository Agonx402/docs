---
title: "Participants"
description: "How participant accounts work in Agon V4."
---

Every wallet that uses Agon registers once as a **participant**.

That participant account binds one wallet to one stable `participant_id`. The mapping does not recycle, which means offchain systems can safely treat `wallet -> participant_id` as stable for the lifetime of the deployment.

## What A Participant Account Stores

The participant account is the protocol's long-lived identity record.

| Field | Meaning |
| --- | --- |
| `owner` | The wallet that controls the participant |
| `participant_id` | The compact numeric id used inside signed messages and lane seeds |
| `token_balances` | One balance record per allowlisted token used by this participant |
| `inbound_channel_policy` | Whether other parties may open lanes to this participant |
| `bump` | PDA bump for canonical address checks |

The `token_balances` vector is where the protocol keeps available balance, pending withdrawal state, and withdrawal destination data for each token.

## Why Participants Are Permanent

Earlier designs allowed participants to close and recreate their accounts. V4 removed that behavior.

That change made several things easier:

1. `participant_id` stays stable
2. offchain services can index users without worrying about id reuse
3. replay protection gets simpler
4. the docs and mental model get simpler

If a team really wants a new protocol identity, it should use a new wallet. Agon no longer tries to recycle participant ids inside one deployment.

## Inbound Lane Policy

Every participant chooses how other parties may create lanes to it:

| Policy | What it means |
| --- | --- |
| `Permissionless` | Anyone may create a lane to this participant |
| `ConsentRequired` | The payee must sign lane creation |
| `Disabled` | New inbound lanes are rejected |

The default is `ConsentRequired`. That is the safest default for public integrations because it prevents unsolicited lane creation without the payee's involvement.

## Registration Flow

The registration path is intentionally small:

```ts
await program.methods
  .initializeParticipant()
  .accounts({
    owner: wallet.publicKey,
    feeRecipient,
  } as any)
  .signers([wallet])
  .rpc();
```

After registration, you can derive the canonical participant PDA from the wallet address and fetch the participant account whenever you need the stable `participant_id`.

## What Participants Do Not Do

Participants are identities and balance containers. They are not lanes, invoices, or operator accounts by themselves.

The participant account does not say who you are paying. That part lives in `ChannelState`. It also does not store the latest signed commitment messages. Those stay offchain until settlement.
