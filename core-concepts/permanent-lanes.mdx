---
title: "Permanent Lanes"
description: "How Agon V4 models one long-lived payment relationship."
---

In Agon V4, a lane is one one-way payment channel from one payer to one payee for one token.

One permanent channel exists for each `payer_id + payee_id + token_id` combination. That channel is stored in one `ChannelState` PDA under the `channel-v2` seed scheme.

## Overview

Agon keeps lanes one-way because the common case is simple:

- one buyer paying one API
- one agent paying one tool
- one service paying one provider

The payer signs a unilateral cumulative message. The payee does not need to co-sign every offchain update.

## What `ChannelState` Stores

| Field | Meaning |
| --- | --- |
| `token_id` | Settlement token for this lane |
| `payer_id` | Participant that owes value on this lane |
| `payee_id` | Participant that receives value on this lane |
| `settled_cumulative` | Highest cumulative amount already settled |
| `locked_balance` | Funds reserved for this lane |
| `authorized_signer` | Key allowed to sign unilateral payment updates |
| `pending_unlock_amount` + `unlock_requested_at` | Timelocked unlock request |
| `pending_authorized_signer` + `authorized_signer_update_requested_at` | Timelocked signer rotation request |

The lane does not store the latest signed message. That stays offchain until settlement.

## Creating A Lane

The payer creates the lane and pays the account rent. The payee may also need to sign, depending on its inbound-channel policy.

```ts
await program.methods
  .createChannel(tokenId, null)
  .accounts({
    tokenRegistry,
    owner: payer.publicKey,
    payerAccount,
    payeeAccount,
    payeeOwner: payee.publicKey,
    channelState,
    systemProgram: SystemProgram.programId,
  } as any)
  .signers([payer, payee])
  .rpc();
```

If the payer passes `null` for `authorized_signer`, the protocol uses the payer wallet as the signing key by default.

## Locked Funds

Locked funds are optional. Use them when the payee wants guaranteed payment capacity on that lane.

If funds are locked to a lane, settlement spends that locked balance first and shared participant balance second.

Unlocking is delayed on purpose:

1. the payer requests an unlock
2. the protocol waits through the timelock
3. the payer executes the unlock and receives the remaining locked amount back into `available_balance`

This gives the payee time to settle recent commitments before the locked funds are released.

## Channel Signing Key

Each lane has one signing key for unilateral commitments. By default, that is the payer wallet, but the payer can choose a different key for that lane.

This key is different from `authorized_settler`:

- `authorized_signer` signs new cumulative payment amounts
- `authorized_settler` may submit a signed commitment onchain

The signing key can change, but not instantly:

1. request a signer change
2. wait through the timelock
3. execute the change

Until the rotation executes, the old signing key stays active. After it executes, only the new signing key can sign new commitments for that lane.

## Why Lanes Are Permanent

Older designs allowed lanes to close and reopen. V4 removed that behavior.

Permanent lanes make the protocol easier to work with:

1. there is no close-and-reopen replay boundary
2. payment channels stay stable for offchain services
3. the protocol has fewer lifecycle rules to handle

The tradeoff is more long-lived onchain state. V4 accepts that tradeoff because the simpler semantics are worth it for machine-payment integrations.
